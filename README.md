# 코어뱅킹 시뮬레이터 (Bank React)

> 실제 시중은행의 코어뱅킹 업무 흐름을 프론트엔드 단에서 재현하는 **개인 풀스택 학습/포트폴리오 프로젝트**.
> 단순 CRUD 어드민이 아니라 **인증등급, FDS, AML/STR/CTR, DSR, 감사로그, AI 에이전트 거버넌스**까지 포함한 규제 친화 코어뱅킹을 목표로 한다.

## 1. 이 프로젝트가 보여주려는 것

| 흔한 React 데모 | 이 프로젝트 |
|---|---|
| 카드/차트 + 더미 테이블 | **고객 → 직원 → 컴플라이언스** 3개 레이어를 가진 실제 도메인 모델 |
| API 한두 개 mock | 모킹 외부 인터페이스 5종(BOK-Wire / KFTC / CB / CSS / FDS) |
| 단일 권한 | RBAC + ABAC + **직무분리(SoD)** 권한 모델 (8개 ROLE) |
| 콘솔 로그 | **위변조 방지 해시 체인** 감사로그 + L0~L4 위험도 등급 |
| LLM 호출 = 그대로 실행 | **에이전트 액션 위험도 등급 + 사람 승인 큐** |

면접 관점 핵심 메시지: "프레임워크 다룰 줄 안다"가 아니라 **"규제 산업의 도메인을 프론트엔드 설계에 어떻게 반영하는지 안다"**.

## 3. 페이즈 로드맵 (총 45개 화면)

페이즈 분리 원칙은 **"가치 기준 우선순위 + 의존 그래프"**. 페이즈 1·2를 끝내면 *현금이 흐르는 시스템*이 되고, 페이즈 3을 얹으면 *규제·감사가 통과하는 시스템*이 되며, 4·5는 *완성된 은행*이 된다.

### 📐 페이즈 1 — 코어뱅킹 척추 (Core Banking Spine) · 9 화면

> "고객이 로그인 → 자산 조회 → 이체 → 대출 신청·상환의 메인 흐름을 끊김 없이 수행할 수 있는 상태."

| ID | 화면 | 핵심 도메인 포인트 |
|---|---|---|
| CUS-002 | 로그인 | 인증등급 1~4단계, 모킹 OTP, 세션 |
| CUS-004 | 메인 대시보드 | 자산 합계, 마이데이터 위젯, 알림 |
| CUS-005 | 수신 계좌 목록 | 계좌 종류별(자유/적금/외화) |
| CUS-006 | 수신 계좌 상세 / 거래내역 | 5년 거래내역 + **이자지급 시 세금 분리 표시** |
| CUS-007 | 수신 신규 개설 | 인증등급 3 승급, **적합성 진단** |
| CUS-009 | 이체 (자행/타행/예약/외화) | **FDS 5룰 차단·통과·지연**, KFTC 모킹 |
| CUS-013 | 대출 신청 | CB·CSS 모킹, **DSR 실시간 계산 (40% 경고 / 50% 차단)**, 가계대출 **2영업일 숙려** |
| CUS-015 | 대출 상세 / 상환 스케줄 | 원리금균등·체증·만기일시 시뮬레이션 |
| CUS-016 | 대출 상환 | 일부/전액, 중도상환 수수료 |

**완료 정의**: 라우팅·권한 가드 / 모킹 API 13개 / 인증등급 승급 모달 / FDS 시나리오 / 감사로그 적재 / 일배치 BAT-001~003 모킹.

## 4. 공통 영역 — 화면 만들기 전에 정리한 "토대 8종"

화면 단위로 흩어지면 일관성이 깨지는 것들을 먼저 분리해 표준화했다 (`bank-app/화면정의서/00_공통/`).

| 문서 | 다루는 것 |
|---|---|
| `00-01` 시스템 개요 / 아키텍처 | 채널·계정·정보·대외계 매핑 |
| `00-02` **법규 매트릭스** | 12개 법규 코드 ↔ 화면 매핑 |
| `00-03` **사용자 권한 모델** | 8개 ROLE, RBAC + ABAC + 직무분리 |
| `00-04` 공통 영역 (헤더 / 푸터 / **AI 사이드패널 380px**) | 모든 화면 공통 골격 |
| `00-05` **인증·보안 정책** | 인증등급(1~4), 거래별 인증, 세션, 한도, 데이터 보호 |
| `00-06` 사이트맵 / 네비게이션 | 고객/직원 흐름도, 라우팅, IA |
| `00-07` **위험도·감사로그 표준** | L0~L4 액션 등급, 감사로그 표준 필드 |
| `00-08` **외부 인터페이스 모킹** | BOK-Wire, KFTC, CB, CSS, FDS |

각 화면 문서는 13개 표준 섹션 (메타 / 목적 / 와이어프레임 / 컴포넌트 / 입력 / 출력 / 상태 / API / 비즈니스 룰 / 보안·감사 / **에이전트 위험도** / 접근성·반응형 / 검증 시나리오) 으로 작성되어 있다.

## 6. 개발 워크플로우 — "하네스 5단계" (이 프로젝트의 차별점)

작업 1건당 다음 5단계를 강제한다 (`CLAUDE.md` 참조). Git 훅이 단계 위반 시 차단한다.

| Step | 이름 | 게이트 |
|---|---|---|
| 0 | **워크트리 격리** | `git worktree` 자동 생성, 태스크 파일 활성화 |
| 1 | **설계** | `docs/designs/<task-id>.md` 의 "목표" 채움 |
| 2 | **구현 + 단위 테스트** | `evidence/<task-id>/unit/notes.md` 에 검증 기록 |
| 3 | **회귀 테스트** | 무관한 주요 기능 1개 이상 실제 사용 후 기록 |
| 4 | **병합 + 사후 검증** | master 병합 → 재검증 → 커밋 & push |

> 면접 시 이 부분을 강조하면 좋다 — *"기능을 빠르게 짜는 것"* 보다 *"실수를 어디서 막을지 미리 정해놓고 짜는 것"* 을 의식적으로 훈련했다는 신호.
